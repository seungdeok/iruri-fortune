# Architecture Decision Records

## 철학
MVP 속도 최우선. 외부 의존성 최소화. **순수 함수와 결정론(같은 입력 = 같은 결과)을 우선**해서 저장·동기화할 상태를 최소화한다. 백엔드 없이 작동하는 최소 구현을 선택한다.

---

### ADR-001: 백엔드 없는 결정적 운세 엔진
**결정**: 운세를 서버가 아니라 클라이언트에서 순수 함수로 계산한다. 시드 = `hash(birthdate|calendarType|YYYY-MM-DD)`, mulberry32 PRNG로 점수·문구 풀 인덱스를 뽑는다. (`src/fortune/prng.ts`, `src/fortune/engine.ts`)
**이유**: 백엔드 없이도 "같은 사람·같은 날 = 동일 운세, 날짜 변경 = 자동 갱신"을 만들 수 있다. 서버·DB·네트워크가 사라져 배포와 운영이 단순해진다.
**트레이드오프**: 서버 측 커스터마이즈나 실시간 문구 생성이 불가능하다. 문구·행운 아이템은 상수 큐레이션 풀(`src/fortune/data.ts`)에서만 나온다.

### ADR-002: 입력값만 영속화, 운세는 매번 재계산
**결정**: 저장소에는 사용자 입력 `{ birthdate, calendarType }`만 저장하고, 운세 결과는 저장하지 않고 진입 시마다 재계산한다. (`src/storage/userStorage.ts`, `src/hooks/useFortune.ts`)
**이유**: 운세가 결정적이므로 결과를 캐싱할 이유가 없다. 저장하지 않으면 날짜가 바뀌었을 때 스테일한 결과가 남는 문제가 원천 차단된다.
**트레이드오프**: 매 진입마다 재계산하지만 순수 함수라 비용은 무시할 수준이다.

### ADR-003: 라우터 없는 단일 화면 + 4계층 구조
**결정**: 라우터 없이 `App.tsx`가 `ready / user / fortune` 상태에 따라 `LoadingView / BirthInput / FortuneResult`를 전환한다. 코드는 도메인(`fortune/`) ↔ 인프라(`storage/`) ↔ 애플리케이션(`hooks/useFortune`) ↔ 프레젠테이션(`components/`) 4계층으로 분리한다.
**이유**: 화면이 사실상 하나(입력 ↔ 결과)라 라우터는 과한 의존성이다. 계층 분리로 플랫폼 SDK 의존성을 `storage/`에 격리하고 도메인 로직을 순수하게 유지한다.
**트레이드오프**: 화면·플로우가 늘어나면 라우터를 다시 도입해야 한다.
